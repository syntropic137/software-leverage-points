# Subagent Prompt: Template One Software Leverage Point Skill into Target Repo

You are a subagent dispatched by `skill-builder`. Generate ONE repo-specific (L2) skill from a generic (L1) meta-skill, customized for the target repo.

## Inputs

- `SOFTWARE_LEVERAGE_POINT`: which software leverage point to template (e.g., `testing`)
- `L1_SKILL_PATH`: absolute path to the generic SKILL.md to template from
- `TARGET_REPO_PATH`: absolute path to the repo to install into
- `OUTPUT_PATH`: where to write the L2 SKILL.md (e.g., `<TARGET_REPO_PATH>/.claude/skills/<SOFTWARE_LEVERAGE_POINT>/SKILL.md`)

## Workflow

1. Read the L1 SKILL.md at `L1_SKILL_PATH`.
2. Inspect `TARGET_REPO_PATH`: language, frameworks, test layout, build tool, CI config. Use Read/Glob/Grep.
3. Write a new SKILL.md to `OUTPUT_PATH` following the L2 backlink format:
   - Frontmatter: `name: <SOFTWARE_LEVERAGE_POINT>`, `description: Use when reviewing <software-leverage-point> concerns in <repo-name>`
   - First content line, portable across machines: `> **Generic principles:** see L1 meta-skill `/software-leverage-points:<SOFTWARE_LEVERAGE_POINT>`.` Do NOT write `<L1_SKILL_PATH>` or any absolute path; that path is local to the generator's machine and does not resolve in the target repo on a different machine.
   - Second content line, the living-skill banner (emit verbatim, with `YYYY-MM-DD` filled in to today's date): `> **Living skill.** This file is generated by `/software-leverage-points:skill-builder` and reflects repo state as of `YYYY-MM-DD`. Code changes that affect this lens (new ADRs, renamed registries, changed conventions, maturity shifts) should trigger a regeneration. `/software-leverage-points:skill-auditor` watches for drift between this file and current repo state.`
   - `## Repo Context` section with concrete facts inspected from the repo. Prefer durable signals (shape, presence, named conventions) over values that drift (specific version numbers, exact commit counts, last-week deploy stats). Cite a specific value only when the value itself is the calibrating fact.
   - `## Maturity Assessment` section (see template below). Sense the target's stage from signals (CI presence, contributor count, deploy cadence, public consumers, release tags, language/toolchain). The assessment is per-lens, not per-repo: an otherwise-growing repo may legitimately rate at POC for a specific lens.
   - `## Growth Direction` section (see template below). Frame the natural next step for THIS lens at the current maturity stage.
   - `## Repo-Specific Notes` section with conventions, hard local rules, deliberate deferrals, and known gaps tailored to the inspected stack. Phrase items as observations and conventions, NOT as severity-tagged findings. Do NOT use `error` / `warn` / `info` headings or bullet prefixes; severity is assigned at review time by `software-leverage-review`, not by this file. Three useful groupings: "Hard local rules (the repo has already taken a position)", "Conventions in place", "Known gaps the orchestrator should weight against the maturity rating."
   - `## Workflow` (two short paragraphs): (a) the orchestrator reads `## Repo Context` + `## Maturity Assessment` + `## Growth Direction` + `## Repo-Specific Notes` plus the L1, then walks the diff; (b) severity per finding is the orchestrator's call, informed by maturity. Do NOT instruct the L2 to "emit findings"; the L2 is a context document.

The two new sections MUST be emitted verbatim (with the assessed values filled in) between `## Repo Context` and `## Repo-Specific Notes`:

```markdown
## Maturity Assessment

- **Current level:** (POC / growing / stable / legacy / safety-critical)
- **Signals:** (specific facts the assessment is based on, e.g., "no CI, single contributor, deployed only to local dev")
- **Last reviewed:** YYYY-MM-DD

## Growth Direction

- **Natural next step for this lens:** (e.g., "introduce structured logging library; current `print()` is fine for POC scope")
- **Trigger that would force the step:** (e.g., "first incident where log noise hides the cause")
- **Don't bother yet:** (things that would be premature at this maturity)
```

## Output

Confirm the write with the path written. Do not duplicate generic content from L1; only repo-specific facts. Re-read your output before finishing and verify: (1) no `error` / `warn` / `info` labels appear in the body; (2) no absolute filesystem paths appear (the L1 backlink is the namespaced form `/software-leverage-points:<lp>`); (3) Repo Context favours durable signals over values that will drift.
