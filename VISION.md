# Vision

This is the canonical statement of what `software-leverage-points` is for, what success looks like, and how we hold ourselves to it. Read this first when joining the project or when deciding what to build next.

## What this plugin is for

A system of focused review agents that scales the operator out of the triage loop.

Each high-leverage software concern (testing, logging, architecture, dependencies, security, etc.) is its own specialist agent. Specialists run in parallel against a target (a plan, a PR, a codebase), produce lens-specific findings, and feed an aggregator that surfaces the headline a maintainer can act on. The pattern repeats across review surfaces: plan-time review, PR-time review, and full-codebase audits all use the same fan-out shape.

## The end state

The operator runs as a product manager. They name features, decide trade-offs, choose what ships. The review system absorbs the layer below: triage, prioritization, "is this finding real," "what shape does the fix take." When the system surfaces a headline, the operator makes a product call. When it doesn't, the operator trusts the silence.

This is moving judgment up the stack. Routine reviewer-grade decisions are absorbed by the system. Product-grade decisions stay with the operator.

**Trust is the load-bearing property.** Everything else serves trust: convergence-as-evidence, cluster integrity, calibration against real ADRs, predictable severity assignment. If trust breaks, the operator re-enters the loop and the system's value collapses.

## The three-layer pattern

Three layers, three jobs.

**L1 (generic meta-skills).** Hand-authored, repo-agnostic. Statements of "what good looks like" for one leverage point: principles, red flags, citations to books and named practitioners. L1 ships in this plugin. Set in stone in spirit; updated over time as understanding deepens.

**L2 (repo-specific calibration).** Machine-generated by `skill-builder` reading L1 and inspecting a target repo. Records how the L1 principles land here: maturity, conventions, deferred decisions, gaps. L2 is a context document, not a reviewer. L2 contains no severity language; severity is assigned only at review time by the orchestrator.

**Orchestrator (review-time runtime).** `software-leverage-review` composes L1 + L2 + a target, fans out parallel specialists, deduplicates and synthesizes findings, and produces a structured output organized by the severity-action policy below.

L1 and L2 are also useful to in-the-loop agents outside the review flow. An implementation agent inheriting the L2 calibration ("this repo's testing scope is per ADR-003," "the secret loader is `scripts/config/env.ts`") writes better code without the operator re-explaining the conventions.

## The five surfaces this plugin serves

| Surface | Purpose | Operator |
|---|---|---|
| **L2 bootstrap** | First-time calibration of a new repo. Produces 17 L2 SKILL.md files. | `skill-builder` |
| **L2 maintenance** | Detect drift between L2 claims and current repo state; trigger partial regeneration of stale L2s. | `skill-auditor` (with optional change-context input) + `skill-builder` partial regen |
| **Plan review** | Pre-implementation review of a plan document. | `software-leverage-review` |
| **PR / diff review** | Implementation review of a PR or commit-to-commit diff. | `software-leverage-review` |
| **Codebase sweep** | Full-repo audit producing PR-sized incremental fix plans. | `software-leverage-review` |

L2 maintenance is two operators composed, not a new operator skill. `skill-auditor` accepts a `change_context` input (a diff, a list of changed files, or a free-text summary) for scoped audits. `skill-builder` already accepts a `software_leverage_point_subset` input for partial regeneration of only the drifted lenses.

## Severity-action policy

The canonical encoding lives at [`skills/software-leverage-review/severity-action-policy.md`](skills/software-leverage-review/severity-action-policy.md). It defines the severity enum, the effort enum, the action matrix that routes each finding to one of three actions (auto-fix / promote / bulk), and the output structure that maintainers read.

The orchestrator's prompts and the output schema reference that file directly. Do not duplicate the policy here; edit the canonical and `just qa` will surface any consumer that drifts.

The shape, in one sentence: severity decides whether the system attempts a fix; effort decides whether the system has the authority to ship without operator review (any `effort: large` always promotes to the operator); low and info are bulked into one audit-only section. Quality is the first-order constraint; cost and speed are optimization targets we tune via evals once quality is locked.

## The end-to-end workflow shape

The plugin's review pipelines are designed to be invoked as workflow phases inside an outer system (Syntropic137's workflow engine is the primary integration target). The end-state shape:

```
Operator input: feature spec
  │
  ├─ Phase 1: Plan
  │     Outputs: plan document
  │
  ├─ Phase 2: Plan review                        ← software-leverage-review against plan
  │     Outputs: review with severity-action policy applied
  │
  ├─ Phase 3: Plan revision
  │     Inputs: plan + auto-handled findings
  │     Outputs: revised plan with error/warn-now items addressed
  │
  ├─ Phase 4: Implementation
  │     Outputs: PR
  │
  ├─ Phase 5: PR review                          ← software-leverage-review against diff
  │     Outputs: review with severity-action policy applied
  │
  └─ Phase 6: PR revision
        Inputs: PR + auto-handled findings
        Outputs: PR amended with fixes; ready to merge
```

Operator interactions are reduced to: feature spec at the start, ship-or-defer decisions when the system explicitly escalates, product-level trade-offs.

## Acceptance tests

Each test is testable: given inputs, the system produces outputs that meet a quality gate. Tests are versioned together with the codebase; they pass or fail and are tracked in `docs/evals/phase-3-progress.md` (or its successor).

### AT-1: L2 bootstrap

**Given** a target repo with foundational docs and a structured layout,
**When** `skill-builder` runs,
**Then:**

1. 17 L2 SKILL.md files emitted to `<target>/.claude/skills/<lp>/SKILL.md`.
2. Each L2 cites at least one specific repo artifact (file path, ADR number, named convention).
3. `## Maturity Assessment` names concrete signals; `## Growth Direction` includes an observable trigger.
4. `## Repo-Specific Notes` is observation-shaped, not severity-tagged.
5. L1 backlink uses the namespaced form `/software-leverage-points:<lp>`, not an absolute path.
6. Living-skill banner present with generation date.
7. No `error / warn / info` labels in any L2 body.
8. Wall under 10 minutes; cost under $15.

### AT-2: L2 maintenance (auditor + partial regen)

**Given** a target repo whose L2 skills are stale relative to current code,
**When** `skill-auditor` runs (optionally with `change_context` input naming changed files or a diff), then `skill-builder` runs with `software_leverage_point_subset` set to the drifted lenses,
**Then:**

1. Auditor surfaces a list of drifted L2s with specific drift findings (claimed value vs actual value, citing the file or signal).
2. With `change_context`, the audit is scoped: only L2s touching the changed surface are checked; full audit otherwise.
3. Partial regeneration touches only the drifted L2s; their `Last reviewed` date updates.
4. Cost scales with drifted lens count, not total lens count.
5. Wall under 5 minutes for a partial sync.

This composed flow replaces a separate "sync pipeline." Both operators already exist; the open work is `change_context` as an optional input to `skill-auditor`.

### AT-3: Plan review

**Given** a plan document and a target repo with current L2 skills,
**When** the orchestrator runs against the plan,
**Then:**

1. All 17 lenses return findings, or explicit "no concerns through this lens."
2. Findings are deduplicated; cluster integrity survives manual audit (each cluster's `original_indices` describe the same observation).
3. Output follows the severity-action policy: headline (auto-handled), promoted (human review for large-effort items), bulk (low/info audit-only).
4. At least one auto-handled headline finding has enough detail that a plan-revision agent incorporates the fix without operator input.
5. Promoted items each include the rationale for promotion (large effort, scope decision needed).
6. Wall under 15 minutes; cost under $25.

### AT-4: PR / diff review

**Given** a PR or diff base..head ref pair,
**When** the orchestrator runs against the diff,
**Then:**

1. All 17 lenses return findings.
2. Severity calibration matches PR-stage maturity (implementation review weights differ from plan review).
3. Findings reference diff hunks (file:line) where applicable.
4. Output follows the severity-action policy: headline / promoted / bulk.
5. Auto-handled findings are amend-able to the PR by an implementation agent without operator authoring code.
6. Promoted findings (large-effort) are surfaced for operator decision before the PR-revision phase proceeds.
7. Wall under 10 minutes; cost under $15.

### AT-5: Codebase sweep

**Given** target repo HEAD and current L2 skills,
**When** the orchestrator runs against the full codebase,
**Then:**

1. Findings deduplicate to between 30 and 100 distinct issues.
2. At least 5 findings have lens_count >= 3.
3. Output is grouped into PR-sized increments organized into 4 waves (Foundation / Quick wins / Structural / Promoted-for-review).
4. Each auto-fix increment has a PR description draft and acceptance criteria.
5. Wave 1 increments unblock subsequent waves.
6. Large-effort increments land in the Promoted wave with rationale; operator decides whether to split, schedule, or ship as-is.
7. Low-priority items are bulked into one section.
8. Wall under 20 minutes; cost under $30.

### AT-6: Trust without verification

**Given** a fully-deduped review output from any pipeline,
**When** the operator reads only the headline section,
**Then:**

1. Headline findings are concrete (file:line, ADR ref, named convention).
2. Each headline finding's suggested fix is actionable.
3. No headline finding is invalid (false positive) on manual spot-check of 3 random items.
4. Operator's required action per headline finding is one of: ship, defer with reason, or escalate. Never "investigate to see if it's real."

This is the trust trip-wire. If AT-6 fails, the system has not earned the right to take the operator out of the loop, and quality must be addressed before any further cost optimization.

### AT-7: Workflow integration

**Given** an outer workflow engine (Syntropic137 or equivalent) and a feature request,
**When** a workflow runs through plan -> plan-review -> plan-revision -> implementation -> PR-review -> PR-revision phases,
**Then:**

1. Each review phase invokes this plugin's orchestrator with the appropriate target type.
2. Auto-handled findings are incorporated by the revision phases without operator input.
3. Bulk findings are surfaced in the workflow output.
4. Operator interactions are: feature spec, ship/defer/escalate decisions, product-level trade-offs.
5. Workflow is interruptible: operator can pause at any review phase and re-enter manually.

## Non-goals

- Replacing the operator's product judgment.
- Competing with single-pass LLM review tools. The differentiator is parallel specialists with calibrated aggregation.
- A general-purpose code-fix agent. The orchestrator surfaces what to fix; downstream phases (plan-revision, PR-revision) do the fixing.
- A web service. CLI plugin first; web layer sits on top of the orchestrator if added later.
- Multi-vendor parity at v1. Claude Code is the v1 target; Codex / Gemini / Cursor are deferred.

## Discipline

Trust takes one bad review to lose and many good reviews to rebuild. Build accordingly:

- Quality is a hard constraint, not a target. Ship a slow, expensive review you trust before a fast, cheap one you don't.
- Vision changes update this doc first, then the code. Not the other way around.
- Each acceptance test gets a status row in the eval-progress tracker. Tests stay current; if reality outgrows a test, retire it explicitly rather than letting it lie.
- Re-read this doc at the start of each work session before deciding what to build next.

## How this connects to the broader pattern

The L1 / L2 / orchestrator shape is reusable beyond software review. Any domain that decomposes into independent specialist concerns calibrated to a target (compliance audits, security threat modeling, design reviews, ops runbooks, regulatory filings) fits the same architecture. Software review is the first instance; ADR-0001 captures the architectural decision and its generalization.
