# ADR-0001: Three-layer skill pattern (generic meta-skill, repo-specific skill, parallel-fan-out orchestrator)

- **Status:** Accepted
- **Date:** 2026-04-28
- **Decision-makers:** plugin maintainers
- **Related:** README "What lives in each layer", CLAUDE.md project rule #5

## Context

Reviewing software through high-leverage lenses (testing discipline, logging discipline, dependency hygiene, security boundaries, etc.) is the kind of task that benefits from specialist attention per concern. A single-pass review by one agent tends to wash out cross-cutting issues: the model weighs everything at once, loses focus, and produces findings biased toward whichever lens fired most strongly.

The natural decomposition is one specialist per concern: a "testing reviewer," a "logging reviewer," a "dependency reviewer," and so on. Run them in parallel, then synthesize.

That decomposition raises a structural question: **where do the lens definitions live, and how do they get calibrated to a specific repo?**

The naive options:

1. **One generic skill per lens, no repo customization.** Every review uses the same generic checklist. Fast to ship, but the findings are repo-blind: a TODO comment is `error` in a production codebase and `info` in a POC; a generic checklist cannot distinguish.
2. **One hand-authored skill per (repo, lens) pair.** Maximally calibrated, but does not scale: 20 repos times 17 lenses is 340 hand-authored skills, and they drift the moment the repo changes.
3. **Three layers: a generic L1 per lens, a generated L2 per (repo, lens) pair, and an orchestrator that composes them at review time.** L1 is hand-authored and stable. L2 is machine-generated by inspecting the target repo. The orchestrator runs at review time and assigns severity using L2's calibration.

This ADR documents option 3 and its consequences.

The pattern emerged from observing that:

- The 17 leverage-point reviewer skills, fanned out in parallel against a single target, produce 17 lens-specific findings in roughly the wall-clock time of one slowest reviewer (~5 minutes).
- The same fan-out at L2 generation time (`skill-builder` dispatching 17 sub-skill generators in parallel) produces 17 calibrated repo-specific skills in the same ~5 minutes.
- The economic cost is real but defensible (~$10/repo for L2 bootstrap on Opus, with ~91% cache-read amortization), and the value is asymmetric: a one-time cost for a context document that downstream reviews use repeatedly.

The pattern generalizes beyond this plugin. Any concern that can be expressed as "a list of independent specialist questions, calibrated to a specific target, run in parallel" fits the same shape.

## Decision

We adopt a three-layer structure with explicit responsibility per layer.

### Layer 1: Generic meta-skill (L1)

**What it is.** A hand-authored skill in this plugin's `skills/<lp>/SKILL.md`. Repo-agnostic. States "what good looks like" for one leverage point: principles, red flags, references to books and named practitioners. Cites canonical sources (Beck, Fowler, Khorikov, Ousterhout, etc.) so the reasoning is grounded.

**What it is NOT.** L1 does not know about any specific repo. It does not contain repo paths, ADR numbers, version pins, or any signal that would not survive copying to another repo.

**Lifecycle.** Versioned with the plugin. Updated when our understanding of the lens evolves. Subject to the project rules in CLAUDE.md (no em-dashes, cite sources, behavior-shaping content not reworded without eval evidence).

### Layer 2: Repo-specific skill (L2)

**What it is.** A machine-generated skill at the target repo's `.claude/skills/<lp>/SKILL.md`. A *context document* about how the L1 principles land in this specific repo: what conventions are in place, what the maturity rating is, what is deliberately deferred, what gaps are known.

**Required sections** (enforced by `skill-builder/SKILL.md` and `skill-builder/prompt_template-one-skill.md`):

- Frontmatter (`name`, `description`) where description specifies "in this repo" so autonomous skill discovery picks the L2 over the L1 when reviewing the local repo.
- L1 backlink in the portable namespaced form `/software-leverage-points:<lp>` (not absolute filesystem path).
- `## Repo Context`: facts inspected from the target. Prefer durable signals over values that drift.
- `## Maturity Assessment`: per-lens (not per-repo) rating with concrete signals and a `Last reviewed` date.
- `## Growth Direction`: natural next step + trigger + don't-bother-yet.
- `## Repo-Specific Notes`: local conventions, hard rules, deliberate deferrals, known gaps. Phrased as observations, NOT as severity-tagged findings.
- `## Workflow`: a short note that the orchestrator reads this file plus L1 plus a target, and that severity is the orchestrator's call.

**What it is NOT.** L2 does not emit findings. L2 does not contain `error` / `warn` / `info` severity labels. L2 is not a self-contained reviewer.

**Lifecycle.** Generated by `skill-builder` from L1 plus repo inspection. Drift is expected (the repo changes; the L2 staleness grows). `skill-auditor` is the safety net: it reads the `Last reviewed` date and compares L2 claims against current repo state, flagging drift for re-generation.

### Layer 3: Orchestrator (review-time runtime)

**What it is.** `software-leverage-review`, a skill that at review time composes L1 (principles) + L2 (repo context) + a target (plan, PR, or diff), fans out parallel subagents (one per lens), and produces severity-tagged findings.

**Severity ownership rule.** Severity (`error` / `warn` / `info`) is assigned only here. The orchestrator uses the L2's maturity rating as a calibration input: the same observation may yield `error` against a `stable` repo and `info` against a `POC` repo. This is the core of the pattern: L2 records the observation, the orchestrator picks the severity at review time when the calibration context is fully resolved.

**Fan-out shape.** N parallel subagents, where N is the number of relevant lenses (currently 17 for this plugin). Wall-clock is bounded by the slowest subagent, not by the number of lenses. The orchestrator post-processes the fan-out output: dedup overlap, surface cross-cutting patterns, sort by severity, write a single artifact.

### Generation step (the bridge)

`skill-builder` is the operator that bridges L1 to L2. It reads each L1, inspects the target repo, and writes a calibrated L2. It is itself a fan-out: the parent dispatches N sub-skill-generation subagents in parallel, one per lens.

The generation step is its own pattern instance: parallel specialists (one per lens) producing parallel artifacts. The same shape as the orchestrator, but at a different lifecycle phase (one-time per repo, vs every review).

## Consequences

### What this enables

- **Parallel specialist review.** The 17 lenses run in parallel; total wall time is roughly the slowest single review, not the sum. The Phase 3 Stage 1 eval observed ~5 minutes for 17 parallel L2 generations against a single target.
- **Repo-calibrated findings without per-repo hand-authoring.** L2 generation costs ~$10 per target on Opus (one-time). Subsequent reviews against that L2 are cheap because the calibration is already in place.
- **A clean severity contract.** Reviewers downstream can trust that any `error` flag came from the orchestrator, with full L1 + L2 context in scope. They never have to wonder whether an L2 author over- or under-weighted a finding.
- **Drift management as a separate concern.** Because L2 is a context document rather than a reviewer, drift detection is a documentation problem (`skill-auditor` reads `Last reviewed` and signals) rather than a behavior problem.
- **Generalization.** The pattern is reusable wherever the structure "many independent specialists, calibrated to a target, run in parallel" applies. Plausible adjacents: per-language code-review checklists, per-domain compliance audits, per-tool migration plans.

### What this costs

- **L2 generation is not free.** ~$10 per target per round on Opus, ~5 minutes wall. Cheaper at lower model tiers, but quality of inspection drops. The cost is a one-time tax per repo; ongoing reviews amortize against it.
- **L2 drift is a real failure mode.** A six-month-old L2 will reference renamed files, deleted ADRs, or stale version pins. `skill-auditor` mitigates but does not eliminate. The mitigation is to prefer durable signals (shape, presence, named conventions) over drifting values (specific versions, exact counts) when generating L2.
- **Two skills with the same lens name coexist.** When the plugin is installed and a target repo has L2s, both `/software-leverage-points:testing` (L1) and the project-scoped `testing` (L2) are visible. Disambiguation relies on the L2's `description` mentioning "in this repo" so autonomous skill discovery picks the right one. This is a design feature, not a bug, but it requires the generator to consistently produce repo-scoped descriptions.
- **The orchestrator's severity calibration is now the load-bearing part of the system.** If the orchestrator weights observations wrongly, every review is wrong. Severity assignment cannot be tested against a fixture without exercising the full L1 + L2 + target composition.

### What this rules out

- **L2 cannot be used standalone as a reviewer.** Anyone tempted to invoke `/testing` directly inside a target repo and expect findings will get a context document, not findings. This is documented in CLAUDE.md project rule #5 and in README's "What lives in each layer" section.
- **Severity language outside the orchestrator is forbidden.** L1 may discuss what kinds of issues are typically critical, but does not assign severity to a specific change. L2 records observations, not findings. Only the orchestrator emits severity.

## Validation evidence

The pattern was validated by Phase 3 Stage 1 of the eval plan (`docs/evals/eval-plan-phase-3-2026-04-28.md`):

- Two targets (`openclaw-hermes`, `syntropic137`) bootstrapped in parallel sessions.
- 17/17 L2 SKILL.md files emitted per target, zero errors.
- Per-target wall clock: ~5 minutes. Per-target cost: ~$10. Combined: $19.50 for 34 calibrated skills.
- First-round outputs scored 80% pass on user review: repo-grounding axis was strong (real ADR references, named files, repo-specific conventions); two structural axes failed uniformly (severity emission leaked into L2 body, absolute filesystem paths leaked into L1 backlink).
- Round 2 (after generator patches) is the validation that the corrected mental model produces correctly-shaped L2s.

## Generalization: this is a skill pattern, not just a plugin

The L1 / L2 / orchestrator decomposition is reusable wherever an agentic workflow has the shape "N independent specialists, target-calibrated, parallelizable." Applying it to a new domain requires:

1. **Define the lenses.** Enumerate the independent concerns. Each becomes one L1.
2. **Define the calibration shape.** What facts about the target should L2 record? Maturity is one axis here; in another domain it might be regulatory tier, size, or risk profile.
3. **Define the orchestrator's severity calibration.** Given the L2's calibration shape, how does the orchestrator translate observations into severity?
4. **Define the generator.** What does `skill-builder`'s analogue inspect, and what shape does it emit?

In all cases, the load-bearing rule from this ADR carries: the layer that emits the final judgment (severity, score, recommendation) is the only layer where that judgment lives. Lower layers record observations; the top layer composes and judges.

## Mechanical enforcement (added 2026-04-29)

The three-layer pattern depends on three contracts that must not silently drift. Each is now mechanically enforced by `just qa`:

### 1. Severity / action policy is single-sourced

The canonical encoding lives at `skills/software-leverage-review/severity-action-policy.md` (severity enum, effort enum, action matrix, maturity calibration, output structures for both stages). Downstream consumers (output schemas, orchestrator skill, per-SLP prompt) reference it; they do not duplicate. The `scripts/check-policy-consistency.sh` validator (wired into `just qa` as `check-policy`) verifies that:

- Every consumer references `severity-action-policy.md`.
- Schema enum values mirror the canonical exactly.
- Legacy enum values (`error`, `warn`, `now`, `next`, `someday`, `priority`) do not appear.

This means a future operator can rename `critical` to `blocker` by editing the canonical alone; QA names every consumer file that still references the old value.

### 2. SLP membership is decoupled from filesystem layout

The orchestrator's fan-out iterates over an auto-generated manifest at `skills/software-leverage-review/slp-manifest.yaml`, **not** over a filesystem listing of sibling directories. Without this, any unrelated skill installed in the same scope (a different plugin, a project-local skill, a globally-installed skill) would be picked up as an SLP and dispatched as a lens reviewer. The manifest is the safety boundary that prevents accidental scope creep.

The manifest is regenerated by `scripts/regenerate-catalogs.sh` from the SLP skill directories themselves. `just qa` runs the regeneration and `git diff --exit-code` on the manifest output, so any drift between filesystem state and committed manifest fails the build.

### 3. Output stages are schema-distinct

Stage 1 (per-SLP output) and Stage 2 (orchestrator merged output) have separate schema files (`slp-output-schema.yaml` and `orchestrator-output-schema.yaml`) sharing common types via `common-types.yaml`. Stage 1 findings have severity + effort but no `action`; Stage 2 findings carry the routed `action` plus `software_leverage_point`. The `validate-output.ts` script can be invoked with `--shape lens|orchestrator` to enforce the right shape, including rejecting Stage 1 outputs that contain an `action` field (a category error: action belongs to the orchestrator, not the SLP subagent).

These three contracts together are what make the pattern trustworthy at scale: the operator can rely on the system's output without re-verifying that each layer obeyed its constraints, because mechanical checks already did.

## Performance architecture (added 2026-04-29, evidence: EVAL-001)

The orchestrator's runtime shape is itself a load-bearing decision. EVAL-001 ([`docs/evals/EVAL-001-orchestrator-perf.md`](../evals/EVAL-001-orchestrator-perf.md)) ran a five-cell experiment (baseline + Phase A+B + three dedup-strategy variants) and produced four rules that now constrain the orchestrator implementation. Changes that violate these rules require a new ADR or an amendment.

### 1. Stage 1 fan-out is parallel and bounded by the slowest SLP

The 17 SLP reviewer subagents are dispatched in a single assistant message, all with `model: "opus"`. Wall time is bounded by the slowest individual subagent (~3 minutes against a representative plan), not by the count of SLPs. Sequential fan-out is forbidden.

### 2. Stage 1 outputs are written to per-SLP scratch files, never returned in subagent response bodies

Each SLP subagent receives an `OUTPUT_FILE_PATH` parameter and uses the Write tool to emit its findings YAML directly to that file. The subagent's response is a one-line confirmation. The parent does not re-ingest 17 response bodies; a Bash concat or a Python aggregator stitches per-SLP files for the dedup pass. This eliminates the ~7-minute parent re-ingestion overhead measured in the pre-EVAL-001 baseline.

### 3. Dedup is a single-shot Opus call

Sonnet dedup, action-sharded parallel Opus dedup, and mechanical-pre-cluster + Sonnet-polish were all tested in EVAL-001's Phase C matrix. All three regressed on cluster integrity (Sonnet collapsed the effort dimension; mechanical clustering missed cross-SLP vocabulary differences and kept duplicates as separate clusters; action-sharded did not finish in acceptable time). Single Opus dedup at ~6 minutes is the floor for cluster-integrity quality at this stage.

### 4. Routing, presentation, and plan-v2 generation are deterministic Python

The action matrix (Step 5), three-section markdown rendering (Step 8), and plan-v2 generation are all mechanical transforms with no LLM judgment required. They are implemented as Python scripts under `skills/software-leverage-review/scripts/` (`render-review.py`, `build-plan-v2.py`). LLM calls at these stages introduce schema drift (invented sections, malformed enum values, dropped findings) without compensating quality benefit, as documented in EVAL-001.

These four rules together produce the ~11.5-minute wall budget for a plan-review run (Stage 1 ~3min + dedup ~6min + Python steps <30s). Future optimization should target the Stage 1 floor (e.g., Sonnet at Stage 1 to compress the dominant cost) rather than the post-Stage-1 pipeline, which is now near-optimal.
