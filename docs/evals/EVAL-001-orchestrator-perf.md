# 2026-04-29: Orchestrator performance and dedup-strategy matrix

**Scope:** wall-time and cost optimization for `software-leverage-review` runs against a real plan target. Five runs in total: a baseline, a Phase A+B validation, and a three-cell dedup-strategy matrix.

**What was held constant:**

- Stage 1: 17 SLP reviewers fanned out in parallel, each on Opus 4.7
- Target: a plan document (~680 lines) for a real-feature integration in a target repo
- L2 calibration: pre-existing repo-specific L2 SKILL.md files generated by `skill-builder`
- Presentation: deterministic Python (`scripts/render-review.py`) for the three-section markdown report
- Plan-v2 generation: deterministic Python

**What varied:** the dedup pass model and strategy.

## Headline results

| Run | Dedup | Wall | Cost | Deduped | Headline / Promoted / Bulk |
|---|---|---:|---:|---:|---:|
| Baseline (Run 1) | Opus, single call | 20m 34s | $12.03 | 72 | 48 / 2 / 22 |
| Phase A+B validation (Run 3) | Opus, single call | **11m 36s** | $11.95 | 60 | 39 / 1 / 20 |
| C1: Sonnet dedup | Sonnet, single call | 12m 45s | $11.82 | 101 | 69 / 0 / 32 |
| C2: action-sharded | Opus, 3 parallel + reconcile | killed (~14m partial) | n/a | n/a | n/a |
| C3: mechanical + Sonnet polish | Python pre-cluster + Sonnet polish | 7m 42s | $11.28 | 118 | 76 / 2 / 40 |

## Phase A+B: the surprise that landed (Run 3)

**Phase A** moved each SLP subagent's findings from the response body to a dedicated file written via the Write tool (`raw-findings/<slp>.yaml`). The orchestrator no longer re-ingests 17 subagent responses to assemble a concatenated input for dedup; a Bash script stitches the per-SLP files instead.

**Phase B** replaced the Haiku presentation pass with a deterministic Python templater that maps `action -> section` and emits the three-section markdown.

**Result:** wall dropped from 20m 34s to 11m 36s (-44%) at flat cost. Quality preserved (top headline findings overlap with baseline). The 17 per-SLP files all wrote successfully.

**Forensics insight that drove this work.** A pre-Phase-A run-log analysis revealed that the parent's "re-ingest the 17 subagent responses to write raw-findings.yaml" step cost ~7 minutes (41% of post-Stage-1 time). It was pure orchestration overhead with no LLM judgment. Phase A eliminated it. The Haiku presentation, by comparison, was only 30s of the post-Stage-1 budget; replacing it with Python was for quality (the Haiku step had invented a Promoted item, malformed a severity field, and dropped two bulk items in earlier runs) more than for speed.

## C1 Sonnet dedup: regressed on every dimension

C1 swapped Opus dedup for Sonnet dedup, holding everything else constant. Wall went up (12m 45s vs Run 3's 11m 36s); compression got weaker (101 clusters vs 60); the Promoted bucket was empty (Sonnet collapsed the effort dimension during dedup, so no finding came back as `effort: large` and the action matrix routed nothing to `promote`); Sonnet self-reported 63 clusters but the file contained 101, suggesting it did not re-read its own output before responding.

The earlier Phase 3 dedup eval saw the same effort-collapse failure mode in Sonnet. This is a repeating signal: **Sonnet dedup is not a viable substitute for Opus dedup at this stage**. The token-cost saving (~$0.50 per run) is in the noise relative to the $11 Stage 1 floor.

## C3 mechanical pre-cluster: faster wall, much weaker dedup

C3 ran a deterministic Python pre-cluster (file-overlap and token-jaccard) then dispatched parallel Sonnet polish calls per multi-member cluster. Wall dropped to 7m 42s (the matrix's best). But the pre-cluster missed most semantic duplicates: 140 raw findings produced 118 clusters, with 102 singletons and only 16 multi-member clusters. The headline ballooned to 76 items (vs baseline 48) because the same observation, phrased differently by 4-6 different SLPs, stayed in 4-6 separate clusters.

A spot-check of the C3 headline confirmed the issue: items 1, 2, 5, 9, and 12 were all about the same `compose.yaml` env-var registry drift, separated only by SLP-specific terminology (one says "registry-is-SSOT," another says "magic numbers," another says "ADR-002 violation"). Token-overlap clustering cannot bridge cross-SLP vocabulary differences. **Mechanical clustering with surface-text similarity is not a viable substitute for Opus semantic dedup.**

This is the second time we have seen mechanical clustering fail on cross-SLP vocabulary (the earlier H3 topic-id experiment had the same root cause from a different angle). The pattern is durable; future investments should not retry mechanical clustering with the same threshold-tuning approach.

## C2 action-sharded: killed before completion

C2 was the user's proposal: pre-route raw findings into three buckets (would-be-auto-fix / would-be-promote / would-be-bulk) using deterministic Python; dispatch three Opus dedup subagents in parallel, one per bucket; run an Opus reconciliation pass to catch cross-bucket duplicates. Theory: 3x parallel dedup compression with cross-bucket safety net.

Run was canceled at ~14 minutes wall-clock (longer than the baseline). Artifacts wrote (findings.yaml, review.md), but final cost / wall / quality metrics were not captured because the kill landed before the orchestrator finished its metrics step.

**Read:** the action-sharded approach did not deliver the expected wall-time gain. With three parallel Opus calls, the slowest dedup gates the whole stage; if one bucket has many findings that produce a long output, the parallelism does not help. The reconciliation pass adds a serial step on top. **Without a tighter time budget on the per-bucket dedup, this approach does not beat single Opus dedup.**

A more careful next experiment would: (a) bound each bucket dedup with a hard token budget, (b) pre-shard with smaller buckets (e.g. 6 SLP groups instead of 3 action buckets), (c) consider whether reconciliation can be deterministic Python (matching by file + canonical-issue prefix similarity). Not pursuing these now.

## Decision impact

These findings inform the orchestrator design recorded in **ADR-0001** ("Three-layer skill pattern"), specifically the "Mechanical enforcement" subsection added 2026-04-29:

1. **Stage 1 fan-out is parallel and bounded.** Confirmed: 17 Opus subagents finish within ~3 minutes wall when each writes to its own file.
2. **Stage 2 dedup is single-shot Opus, not parallelized or model-substituted.** Confirmed by C1, C2, C3 all regressing on quality or wall.
3. **Stage 1.5 routing and Stage 2 presentation are deterministic Python, not LLM calls.** Confirmed by Phase B (Python presentation) eliminating LLM-induced format drift while shaving ~30s.
4. **Per-SLP scratch files are the canonical Stage 1 output, not response-body YAML.** Confirmed by Phase A (per-SLP file writes) eliminating ~7 min of parent re-ingest.

These four points are now load-bearing rules for the orchestrator. The current orchestrator skill (`skills/software-leverage-review/SKILL.md`) and per-SLP prompt (`prompt_review-one-software-leverage-point.md`) implement them.

## Acceptance test impact

`VISION.md` AT-3 (plan review) and AT-5 (codebase sweep) target wall budgets of 15 min and 20 min respectively. With Phase A+B landed, the plan-review pipeline now runs in ~11.5 minutes against a representative target, satisfying AT-3. AT-5 (codebase sweep) has not been re-run since Phase A+B; the prior Stage 4 codebase run took 8m 20s, but that included the parent re-ingest step that is now eliminated, so a re-run is expected to be faster.

## Open follow-ups

1. **Re-run Stage 4 codebase sweep with Phase A+B** to confirm AT-5 wall budget. Estimated wall ~12 min, cost ~$22.
2. **Stage-1 model substitution experiment.** Stage 1 is the dominant cost ($11+ of $12 total). One run with 17 Sonnet SLPs + Opus dedup + Python presentation would test whether Sonnet at Stage 1 holds finding quality. If yes, the per-run cost drops to ~$3-5 and wall to ~5-7 min. If quality regresses, accept current Stage 1 cost as the floor.
3. **Hardening Stage 1 schema compliance.** A few SLP subagents emitted markdown fences around their YAML in earlier runs (1 of 17 in baseline, 4 of 17 in Sonnet C1). The orchestrator currently tolerates this but does not enforce. A small per-SLP retry-on-drift loop, plus the per-SLP scratch directory, would make this contract mechanical.
